title: Feedback Técnico
---

Cómo dar y recibir feedback técnico de forma efectiva. El feedback bien dado acelera el crecimiento; el mal dado genera fricción innecesaria.

## Por qué importa

El feedback técnico es una de las herramientas más poderosas para el crecimiento de un equipo. Un Dev que recibe feedback claro y frecuente mejora más rápido que uno que trabaja en aislamiento.

El problema es que el feedback técnico mal dado genera defensividad, no aprendizaje. La forma importa tanto como el contenido.

## Principios

**Específico sobre general**: "Este método hace demasiadas cosas" es menos útil que "Este método valida el input, llama a la base de datos y formatea la respuesta. Considera separar esas responsabilidades."

**Sobre el código, no sobre la persona**: "Este código es confuso" vs "Esta variable podría tener un nombre más descriptivo para que sea más fácil de entender."

**Con contexto**: Explicar por qué algo es un problema, no solo que lo es. Eso enseña el criterio, no solo la solución.

**Oportuno**: El feedback sobre un PR es más útil que el feedback sobre código que ya está en producción hace 3 meses.

## Feedback en code review

El code review es el momento más natural para feedback técnico. Ver [Code Review](/handbook/engineering-standards/git/code-review) para el proceso completo.

Regla clave: distinguir entre lo que es bloqueante (debe cambiar) y lo que es una sugerencia (sería mejor, pero no es obligatorio).

## Feedback 1:1

Para feedback más profundo sobre patrones de trabajo o decisiones técnicas recurrentes, el espacio es el 1:1 con el Tech Lead o EM.

El feedback en 1:1 sigue el mismo principio: específico, sobre comportamientos observables, con ejemplos concretos.

## Cómo recibir feedback

- Escuchar para entender, no para responder
- Si algo no está claro, preguntar antes de asumir que es una crítica
- Si no estás de acuerdo, argumentar con datos o razonamiento técnico, no con defensividad
- El feedback es información, no un juicio de valor sobre ti como persona

## Feedback ascendente

El feedback no solo fluye de senior a junior. Los Devs también pueden y deben dar feedback al Tech Lead sobre decisiones técnicas, procesos o comunicación.

El canal para eso es el 1:1, la retrospectiva o directamente si la confianza lo permite. Un equipo donde solo fluye el feedback hacia abajo no aprende.
